## 使用前先知道

这些不是让你自己解决，而是帮你提前知道第一次试跑时可能遇到什么，以及该把哪些信息交给 Agent 继续处理。

| 可能遇到的情况 | 你可以怎么做 |
| --- | --- |
| 本地校验通过，但 AI Hub 第一次运行仍报错 | 把报错截图或日志发给 Agent。结构校验只说明 DSL 文件本身过了已知检查，真实运行还会受模型权限、内容安全、媒体节点结果和平台环境影响。 |
| 需要权限、密钥、素材或存储位置 | 先把可提供的信息告诉 Agent，或在 AI Hub 里完成授权；Agent 会继续把它补进下一版方案。 |
| 第一次结果不完全符合预期 | 这通常是正常的首测信号。直接描述哪里不对，Agent 会判断该改输入、Prompt、工具还是输出格式。 |
| 准备放进真实业务里使用 | 先用一个真实小样例跑通，再根据结果决定是否需要 live QA、权限确认或下一版 DSL。 |

这样做的好处是：即使第一次结果还不完美，我们也能更快判断应该改输入、改 Prompt、改工具选择，还是改最终输出格式。

## 如果效果不满意

你可以直接把运行结果发给 Agent，并按下面任意一种方式描述：

| 你看到的问题 | 你可以这样说 |
| --- | --- |
| 内容方向不对 | “它没有抓住我真正想要的业务目标，应该更偏向……” |
| 风格不对 | “语气/画面/结构不符合我们平时的风格，应该更……” |
| 结构不方便用 | “输出内容我不好复制到日常工作里，希望改成……” |
| 信息缺失 | “结果里少了这些关键信息……” |
| 质量不稳定 | “同样输入下结果有时好有时差，希望增强稳定性。” |
| 工具或运行报错 | “导入/运行时报了这个错：……” |

## 下一步怎么迭代

你给出反馈后，Agent 会先更新本项目的 DSL Discovery Brief，再生成下一版 DSL，例如 `{{next_dsl_filename}}`。你不需要自己理解节点细节，只需要描述结果哪里不符合你的目标。
